# Stress 中文模板

按任务类型选模板，不要一次叠太多压力机制。先选主模式，再补质量要求、失败约束和输出格式。

## 通用升级

```text
[任务]

这份输出会被专业人员审查，并与更强的备选答案直接比较。空泛、敷衍、遗漏关键约束、逻辑松散的内容会立刻显得低水平。

请给出这个任务下最强的一版结果。

要求：
- 具体，不空话
- 覆盖关键约束和边界情况
- 让推理和结论容易检查

不要：
- 猜测或编造事实
- 用套话和泛泛建议充字数
- 跳过不确定点和困难部分

如果仍有不确定性，明确写出，并基于现有信息给出最稳妥的答案。

输出前检查：
- 是否直接回答了原问题
- 是否覆盖了硬约束
- 是否区分了事实、假设和判断
- 是否删掉了最像弱点的内容

输出格式：
[格式]
```

不要空话，不要绕圈子，不要用模糊表述掩盖不确定性。

## 问责压力

```text
[任务]

请给出一版经得起追问的答案。

要求：
- 区分事实、假设、判断
- 关键结论必须有依据
- 重要不确定性必须明确写出

不要把推断写成事实，不要给出没有根据的自信结论。

输出格式：
[格式]
```

## 对抗压力

不要虚张声势，不要回避反方观点，不要把薄弱证据包装成确定结论。

## 高质量代码包

```text
[编码任务]

这份实现会和更强的实现方案直接比较，也会被资深工程师审查。如果关键设计无法自圆其说、存在隐藏回归、边界处理薄弱，结果就会被否掉。

请交付正确、克制、可上线的实现。

要求：
- 除非明确要求，否则保持既有行为
- 处理边界情况、失败路径和接口不匹配
- 明确写出关键假设
- 优先做精确修复，不要无意义扩大改动
- 结果必须容易验证

不要臆造 API，不要跳过校验，不要用复杂度掩盖思路不清。

输出前检查：
- 最先会坏在哪里
- 审查者最先会质疑哪里
- 哪一段改动最像“勉强能跑但不够稳”

输出格式：
- 修改摘要
- 代码或补丁
- 风险与假设
- 验证步骤
```

## 高质量代码评审包

```text
[代码评审任务]

这份评审会被拿去和强 reviewer 的结果比较。如果你漏掉关键 bug、回归风险、错误假设或测试缺口，这份评审就不合格。

请给出高信号评审。

要求：
- 先说最严重的问题
- 按严重性和影响排序
- 每个问题都落到具体行为或代码证据
- 区分已确认问题和待确认问题
- 重点关注正确性、回归风险、安全性和缺失测试

不要用空泛表扬或纯风格评论冲淡问题。

输出格式：
- 发现的问题
- 待确认问题
- 简短总结
```

## 高质量分析包

```text
[分析或决策任务]

假设专业审查者和反对者都在盯着这份分析，专门寻找逻辑漏洞、隐藏假设、证据不足和遗漏的权衡。你会因为结论是否经得住挑战而被评价。

请给出清晰、可辩护的结论。

要求：
- 区分事实、假设、推断
- 指出关键不确定性
- 覆盖主要方案、权衡和反方观点
- 说明结论为什么能从证据中推出
- 说明哪些条件变化会导致结论变化

不要把解释当证据，不要把推测写成定论，不要跳过最强反对意见。

输出格式：
- 结论
- 核心理由
- 权衡与备选方案
- 不确定性
```
